Build per-input processors once, not once per harvester

A single add_kubernetes_metadata processor was being rebuilt for every log file Filebeat tailed. With thousands of Kubernetes container logs that meant thousands of Kubernetes API watchers, and ~13× the memory. This page covers the bug, the surrounding pipeline architecture, and how this branch fixes it.

issue #50376 PR #50611 (draft) related PR #50713 (merged) core file filebeat/channel/runner.go

Intro

The bug

Each filestream harvester (one per file) opens its own pipeline client. The code that builds an input's processors ran inside the per-client path, so every harvester constructed a fresh copy of every processor.

The fix

Build the input's processors once when the input starts, and hand every client a thin run-only view of those shared instances. The input owns their lifecycle and closes them exactly once at shutdown.

The catch

Sharing means a client must not close or re-initialise a processor that siblings still use. The wrapper hides Close and SetPaths; the input does both, once.

Why the upstream matters for this PR

The reload driver explains three design choices that otherwise look arbitrary in runner.go:

Idempotent Stop

RunnerList.Reload calls runner.Stop() on every config change for every removed runner (list.go:119). Stop can therefore arrive more than once → the wrapper guards it with sync.Once.

Forwarding SetStatusReporter

Under Agent, the driver wires per-unit status: runner.(status.WithStatusReporter).SetStatusReporter(...) (list.go:154). The embedded interface hides it, so the wrapper re-exposes it (§7③).

Fail-fast at Create

A failed Create is reported per unit when InputUnitID != "" (list.go:141). Building processors eagerly (§4) turns a bad processor config into a clean FAILED unit (§10).

3 · Root cause — construction in the per-client closure

On main, newCommonConfigEditor returned a closure, and the processor construction lived inside that closure. The closure is the editor — it runs on every ConnectWith, i.e. once per harvester.

filebeat/channel/runner.go — BEFORE (origin/main)

func newCommonConfigEditor(beatInfo beat.Info, cfg *conf.C) (pipetool.ConfigEditor, error) {
    config := commonInputConfig{}
    cfg.Unpack(&config)

    // the returned closure IS the editor — it runs on every ConnectWith
    return func(clientCfg beat.ClientConfig) (beat.ClientConfig, error) {
        var indexProcessor beat.Processor
        if !config.Index.IsEmpty() {
            // ... build add_formatted_index ...
        }

        userProcessors, err := processors.New(config.Processors, beatInfo.Logger) // ⟵ PER HARVESTER
        if err != nil { return clientCfg, err }

        // ... assemble procs, attach to clientCfg.Processing.Processor ...
        return clientCfg, nil
    }, nil
}
The defect processors.New(...) calls each processor's constructor. For add_kubernetes_metadata that constructor starts a Kubernetes watcher. Because the call sits inside the per-client editor, N harvesters ⇒ N watchers ⇒ N×(goroutines + caches + API connections).

4 · This branch's approach — hoist construction out of the closure

The fix: build the processors once, before the editor closure, so every client references the same instances. The closure now only wraps and attaches them.

filebeat/channel/runner.go:180 — build user processors ONCE per input

func newCommonConfigEditor(beatInfo beat.Info, cfg *conf.C) (pipetool.ConfigEditor, *processors.Processors, error) {
    config := commonInputConfig{}
    cfg.Unpack(&config)

    // Build the user-configured processors once per input
    userProcs, err := processors.New(config.Processors, beatInfo.Logger) // ⟵ ONCE
    if err != nil { return nil, nil, err }

    return newConfigEditor(beatInfo, config, userProcs)
}

Note the new return signature: it now also returns the *processors.Processors so the caller (the runner) can own their shutdown.

filebeat/channel/runner.go:203 — assemble the shared list, set paths once

func newConfigEditor(beatInfo beat.Info, config commonInputConfig,
    userProcs *processors.Processors) (pipetool.ConfigEditor, *processors.Processors, error) {

    // index processor (optional) + the user processors form the shared list
    shared := processors.NewList(beatInfo.Logger)
    if indexProc != nil { shared.AddProcessor(indexProc) }
    shared.AddProcessors(*userProcs)

    // Path-aware processors must have paths set BEFORE Run, else SafeProcessor
    // returns ErrPathsNotSet and drops every event. The per-client group can no
    // longer do this (the wrapper hides PathSetter), so we do it once, here.
    if err := shared.SetPaths(beatInfo.Paths); err != nil {
        _ = shared.Close()
        return nil, nil, fmt.Errorf("failed to set paths for input processors: %w", err)
    }

    editor := func(clientCfg beat.ClientConfig) (beat.ClientConfig, error) { /* see next tab */ }
    return editor, shared, nil
}

Splitting newConfigEditor out of newCommonConfigEditor separates "build from the registry" from "wire it up", which lets tests inject processors directly instead of registering a global test plugin.

filebeat/channel/runner.go:242 — the closure now only wraps the shared instances

editor := func(clientCfg beat.ClientConfig) (beat.ClientConfig, error) {
    // ... set meta / fields / service.type ...

    procs := processors.NewList(beatInfo.Logger)
    if indexProc != nil { procs.AddProcessor(sharedProcessor{indexProc}) }
    if lst := clientCfg.Processing.Processor; lst != nil { procs.AddProcessor(lst) }
    for _, p := range userProcs.List { procs.AddProcessor(sharedProcessor{p}) } // ⟵ same instances

    clientCfg.Processing.Processor = procs // flat siblings, not a nested group
    return clientCfg, nil
}

No processors.New here anymore. Every client receives a sharedProcessor{} view of the same underlying processor objects.

5 · The sharedProcessor wrapper — a run-only view

The per-client processing group owns its members' lifecycle: when a harvester's client closes, the group calls Close() on each processor, and at build time it calls SetPaths(). Both are wrong for a shared instance — the first harvester to stop would tear down state the others still need.

filebeat/channel/runner.go:170

// sharedProcessor is a run-only view of an input-owned processor.
// Embedding the beat.Processor interface promotes ONLY Run/String,
// hiding Closer and PathSetter, so a pipeline client closing its own
// processor list cannot tear down — or re-initialise — state still
// used by sibling harvesters.
type sharedProcessor struct {
    beat.Processor   // ⟵ only Run() + String() are promoted
}

var _ beat.Processor = sharedProcessor{}
How it works: interface embedding By embedding the narrow beat.Processor interface (not the concrete type), the wrapper only satisfies Run and String. A type-assertion like p.(processors.Closer) or p.(processors.PathSetter) fails, so the group's close/setpaths loops simply skip it. Closing becomes a safe no-op.

It mirrors how pipeline-global processors are already protected from client close, by wrapping them as a function processor:

libbeat/publisher/processing/default.go:382 — the pattern this branch copies, one level down

  if b.processors != nil {
      // function processor hides implementation of processors.PathSetter
      err := b.processors.SetPaths(b.info.Paths)   // set once
      // Add the global pipeline as a function processor, so clients cannot close it
      processors.add(newProcessor(b.processors.title, b.processors.Run))
  }

Global processors are shared across the whole pipeline; this branch needed the same "shared, clients can't close it" guarantee but scoped to a single input. Same idea, narrower scope.

6 · Lifecycle & ownership — who builds, who closes

Ownership moves from "each client" to "the input's runner." The runner is wrapped so its Stop() releases the shared processors, in the right order, exactly once.

filebeat/channel/runner.go:91 — Create wraps the runner only when there is something to release

func (f *commonSettingsFactory) Create(pipeline beat.PipelineConnector, cfg *conf.C) (cfgfile.Runner, error) {
    editor, sharedProcs, err := newCommonConfigEditor(f.info, cfg)
    // ...
    r, err := f.inner.Create(pipetool.WithClientConfigEdit(pipeline, editor), cfg)
    if err != nil { _ = sharedProcs.Close(); return nil, err }

    if len(sharedProcs.List) == 0 { return r, nil } // nothing extra to release → no wrapper
    return &runnerWithSharedProcessors{Runner: r, procs: sharedProcs}, nil
}

filebeat/channel/runner.go:130 — Stop: drain clients first, THEN release shared procs, once

func (r *runnerWithSharedProcessors) Stop() {
    r.stopOnce.Do(func() {           // idempotent: *input.Runner.Stop is not
        r.Runner.Stop()                 // 1. stop input → all clients drain
        _ = r.procs.Close()             // 2. now safe to close shared procs
    })
}

7 · Three subtleties

① Hiding Close would silently break path-aware processors — so paths move too

Normally the per-client group.SetPaths() initialises path-aware processors (cache, script, path-aware conditionals). Once sharedProcessor hides PathSetter, that call can no longer reach them. If nobody sets paths, SafeProcessor.Run trips ErrPathsNotSet and drops every event:

libbeat/processors/safe_processor.go:66

func (p *SafeProcessor) Run(event *beat.Event) (*beat.Event, error) {
    switch p.state {
    case stateInit:
        if _, ok := p.Processor.(PathSetter); ok {
            return nil, ErrPathsNotSet   // ⟵ would drop events if we forgot
        }
    }
    return p.Processor.Run(event)
}

The fix adds a new list-level helper and calls it once at build time (beat paths are fixed at startup, before any input exists):

libbeat/processors/processor.go:167 — new method

func (procs *Processors) SetPaths(paths *paths.Path) error {
    var errs []error
    for _, p := range procs.List {
        if ps, ok := p.(PathSetter); ok { errs = append(errs, ps.SetPaths(paths)) }
    }
    return errors.Join(errs...)
}

② Shared processors stay flat siblings, not a nested group

Each shared processor is added individually to the client list (procs.AddProcessor(sharedProcessor{p})) rather than as one nested sub-group. This preserves the client group's continue-on-error semantics: processing.group keeps going after a non-fatal processor error, whereas a processors.Processors aborts the chain. Flattening keeps behaviour identical to before. Locked by TestProcessorsForConfigIsFlat.

③ Embedding hides optional methods — so they're forwarded by hand

The runner wrapper embeds the cfgfile.Runner interface, which only promotes Start/Stop/String. Two optional methods that callers discover by runtime type-assertion must be re-exposed explicitly, or features silently break:

MethodWho type-asserts for itWhat breaks if missing
SetStatusReporterlibbeat/cfgfile/list.goElastic-Agent status reporting
SetOnce (OnceSetter)crawler.go:142filebeat --once mode

Both are forwarded in runner.go:139-149, each a no-op if the inner runner doesn't implement it.

8 · Two PRs, two strategies for the same root cause

There are two independent fixes for issue #50376, and both are in the tree.

PR #50713 — opt-in, per processor (merged)

Adds libbeat/processors/shared. A processor opts in by wrapping its constructor; the wrapper de-duplicates by config hash with reference counting:

// add_kubernetes_metadata/kubernetes.go:63
processors.RegisterPlugin("add_kubernetes_metadata",
    shared.New(New))   // same cfg ⇒ same instance

Pros: fixes the immediate OOM everywhere the processor is used (filestream, autodiscover, modules). Cons: each stateful processor must remember to opt in; the underlying "build per client" waste remains for everything else.

PR #50611 — structural, this branch (draft)

Fixes the frequency itself: input-level processors are built once per input in filebeat/channel/runner.go and shared via sharedProcessor, regardless of which processor it is.

// build once, share read-only view
userProcs, _ := processors.New(cfg, log)
for _, p := range userProcs.List {
    procs.AddProcessor(sharedProcessor{p})
}

Pros: no per-processor opt-in; correct by construction for all input-level processors. Cons: touches a shared lifecycle path, so it has to get ownership, paths and close-ordering exactly right (sections 5–7).

How to think about them #50713 is a targeted de-duplication at the constructor; #50611 is the structural "don't construct per client in the first place." They are complementary: #50713 shipped first as a targeted fix; #50611 removes the root cause so future stateful processors are safe without opting in.

10 · One user-visible behavioral change

Because processors are now built eagerly at input creation (not lazily on first client connect), a broken processor config now fails the input at startup instead of degrading it after it has started. This aligns filestream & aws-s3 with every other input type.

x-pack/filebeat/tests/integration/managerV2_test.go:1354 — expectations updated

// filestream and aws-s3 now build their input-level processors eagerly at

Why this is an improvement Fail-fast at startup is more discoverable than a half-running input. It is the same model the other inputs already follow, so this removes an inconsistency rather than adding one.
